# Grok2API 调度稳定性落地方案

> 范围：仅处理本地项目 `E:\www\grok2api-v3-migration`。
>
> 目标：解决“经常报错”和“没有可调度账号”，不改服务器 SUB2，不重做注册体系。
>
> 原则：少改、先稳定、保留现有稳定轮询和已有账号功能。

## 1. 当前基础

本地项目已经有：

- `grok_build`、`grok_web`、`grok_console` Provider。
- Provider 和上游模型路由。
- 账号启用、凭证状态、模型能力和额度过滤。
- OAuth/SSO Token 自动刷新。
- Web、免费、付费和模型额度恢复探测。
- 冷却、熔断、Sticky Session、并发租约和稳定轮询。
- 账号检查和重新授权流程。

当前先不做：

- 不重做账号注册和补充流程。
- 不新增复杂的业务分组体系。
- 不先切换智能性能调度。
- 不把所有 403 都归类成账号问题。
- 不升级 SUB2。

## 2. 只解决四个问题

### 问题一：路由实际没有可用账号

GROK 分组或模型配置能保存，不代表请求一定能找到账号。请求必须同时满足：

```text
Provider 匹配
  + 上游模型匹配
  + 路由已启用
  + 出口可用
  + 账号已启用
  + 凭证有效
  + 账号支持该模型
  + 没有额度、冷却或健康阻塞
```

保存路由和启动服务时增加简单自检，直接提示：

- Provider 不匹配。
- 上游模型不存在。
- 没有绑定账号。
- 账号不支持该模型。
- 所有账号都在冷却、额度耗尽或需要重新授权。

### 问题二：不健康账号仍然被调度

普通调度只允许以下账号进入候选池：

- `enabled = true`。
- 认证状态有效。
- 支持当前上游模型。
- 没有账号+模型权限拒绝。
- 没有账号+模型冷却。
- 没有额度阻塞。
- 没有 Provider、出口或并发限制。

账号+模型问题只影响这个模型，不能影响同一账号的其他模型。

Sticky 账号如果已经不健康，必须自动放弃 Sticky，回退到其他候选账号。

### 问题三：403 被错误归因

简单按下面规则处理：

| 现象 | 处理 |
| --- | --- |
| 明确 `permission_denied` 或无模型权限 | 隔离账号+模型 |
| Token 过期或凭证失效 | 刷新一次，失败则要求重新授权 |
| 明确额度耗尽 | 进入额度恢复等待 |
| `Your request was blocked`，没有账号信息 | 不封账号，按上游或出口问题处理 |
| 多个账号共用同一出口都 403 | 冷却或熔断出口 |
| 同一上游从多个出口都 403 | 冷却或熔断上游 |
| 只有一个账号持续明确失败 | 隔离该账号或账号+模型 |

通用 403 不允许直接批量禁用或删除账号。

### 问题四：失败后不能自动恢复

只补齐三类自动恢复：

1. Token 到期前自动刷新。
2. 配额到达 `ResetAt` 后自动重新探测。
3. 账号+模型被临时隔离后延迟探测，成功后恢复。

自动恢复不包括自动充值、自动续费或修改上游真实额度。

## 3. 简化后的调度流程

```text
请求
  -> 解析 Provider 和上游模型
  -> 加载账号候选池
  -> 过滤明确不可用账号
  -> Sticky 命中则复用可用账号
  -> 否则稳定轮询
  -> 调用上游
  -> 根据完整响应结果记录成功或失败
  -> 失败时按账号、模型、上游、出口范围处理
```

保持现有稳定轮询，不在本阶段引入复杂的延迟评分和权重学习。

失败重试规则：

- 账号级失败：排除当前账号，切换下一个账号。
- 模型级失败：隔离当前账号和模型，切换账号。
- 上游或出口失败：切换出口或冷却上游。
- 已经开始输出 SSE 后：不透明切换账号，避免重复回答。
- 同一个请求内：不重复使用已经失败的账号。

## 4. 自动恢复规则

### 4.1 Token

```text
正常 -> 即将过期 -> 刷新 -> 正常
                    ├-> 临时失败：延迟重试
                    └-> 永久失败：需要重新授权
```

要求：

- 只刷新 Provider 支持刷新的账号。
- 使用已有的刷新锁，避免同一账号重复刷新。
- 临时网络或 5xx 不立即禁用账号。
- `invalid_grant` 等永久错误进入重新授权。

### 4.2 配额

```text
正常 -> 配额耗尽 -> 等待 ResetAt -> 探测
                                      ├-> 恢复：重新加入调度
                                      └-> 未恢复：退避后再探测
```

要求：

- 配额恢复只恢复本地调度资格，不修改上游额度。
- 恢复成功后清除对应额度阻塞和 Sticky 绑定。
- 配额未恢复时不能被普通请求反复尝试。

### 4.3 临时健康失败

```text
可用 -> 失败 -> 临时隔离 -> 探测 -> 可用
                              └-> 失败：延长冷却
```

要求：

- 权限拒绝、模型不可用必须有过期时间。
- 重新授权成功后清理旧健康状态。
- 自动探测不能和正常请求无限并发。

## 5. 必须修复的代码问题

按优先级处理：

1. 历史健康回填增加有效期，只使用最新有效记录，避免旧失败状态永久阻塞账号。
2. 统一 `model_unavailable` 的实时写入、检查应用和重启恢复逻辑。
3. 流式请求只有在 SSE 正常结束后才记录为成功。
4. 通用 403 不写入账号永久健康拒绝状态。
5. 路由保存和服务启动增加 Provider、模型、出口和账号候选自检。
6. 调度失败信息明确区分：没有账号、模型不支持、额度耗尽、冷却、上游失败和出口失败。

## 6. 实施顺序

### 第一步：先能看清楚

- 日志记录请求、Provider、上游模型、选中账号、出口和失败范围。
- 记录候选总数以及账号被排除的原因。
- 不改变现有轮询策略。

### 第二步：修复不该调度的问题

- 修复健康状态过期和历史回填。
- 修复 `model_unavailable` 状态一致性。
- 确保不健康账号不会进入普通候选池。
- 确保 Sticky 失效后能够回退。

### 第三步：修复经常报错的问题

- 修复流式请求成功判定。
- 403 按账号、模型、上游和出口分层。
- 对上游和出口增加简单冷却，不误伤账号池。
- 控制失败重试次数，避免重复请求。

### 第四步：接通自动恢复

- 验证 Token 自动刷新。
- 验证额度到期自动探测和回池。
- 为权限拒绝和模型不可用增加延迟探测。
- 增加重启后恢复测试。

## 7. 验收标准

必须满足：

- 明确不健康的账号不会被普通调度选中。
- 单模型不可用不会影响账号其他模型。
- Sticky 账号失效后能自动切换。
- 403 不会导致全部账号被批量禁用。
- 同一出口故障只影响该出口，不污染账号健康。
- Token 过期前能够自动刷新。
- 配额到期后能够自动探测并恢复调度。
- 账号和上游失败不会造成无限重试。
- SSE 中途失败不会记为成功。
- 调度失败时能看到明确原因，而不是只显示“没有可用账号”。

## 8. 本阶段明确不做

- 不修改注册账号流程。
- 不增加自动充值或自动续费。
- 不引入新的复杂账号分组模型。
- 不先做智能延迟排序。
- 不批量删除账号。
- 不因为一次通用 403 批量重新授权。
- 不升级或修改 SUB2。

本方案完成后，目标不是把系统做复杂，而是让每次请求都能稳定地选到可用账号，并且在失败后知道应该隔离账号、模型、上游还是出口。

## 9. 开发实施和验收状态

本方案已在本地项目完成实现和验证，不涉及服务器或 SUB2：

- 已保留账号启用、认证状态、模型能力、模型配额、服务健康与并发过滤。
- 已增加账号+上游模型健康隔离。健康记录 24 小时失效，请求成功后可立即恢复。
- 普通 `403 Your request was blocked` 不会批量禁用账号；明确的权限或模型错误只隔离对应账号和模型。
- 流式请求只在正常收尾后写入成功健康状态。
- 全部账号需重新授权时保留已启用路由用于诊断，返回 `upstream_reauth_required`，不再误报模型不存在。
- 多 Provider 仍优先使用健康路由，异常账号只用于诊断兜底，不会抢占正常 Provider。
- Token 刷新、配额恢复探测、现有补位机制和重启恢复继续使用现有服务；不增加自动充值或自动续费。
